# Pushover alert debounce: implementation plan

## Scope of this bundle

Add a 20-minute debounce in front of the operator-facing Pushover page so a single failed 5-minute health check no longer pages (and, where Upptime's own auto-delete reaches it, no longer leaves a GitHub issue). The bundle modifies one custom workflow (`.github/workflows/pushover-on-incident.yml`) and adds a release-grade test rehearsal documented in the worklog. No design phase: the change is contained to one workflow file, the upstream constraints are already mapped (see § Structural notes), and there is no plausible alternative shape that warrants documenting.

`autonomous_eligible: false` — the closeout's rehearsals (Task 02 Steps 3–4) include "user confirms in chat: Pushover received / not received" gates that need a human in the loop. `/exec-bundle` runs this bundle in supervised mode by default.

## Preconditions

- The `PUSHOVER_APP_TOKEN` and `PUSHOVER_USER_KEY` repo secrets are already populated (referenced by the existing workflow at `.github/workflows/pushover-on-incident.yml:13-14`).
- `gh` CLI is authenticated against `SynVisions/homelab-status` with permission to label issues — needed only for the closeout-task verification.
- The custom `pushover-on-incident.yml` workflow is NOT regenerated by Upptime's template-update job (verified: no "Do not edit this file directly!" header; commit `ae24ea0` shows it was hand-added).

## Structural notes

**The hard constraint.** Upstream `upptime/uptime-monitor@v1.41.3` opens the GitHub issue *synchronously* on the first failed Uptime CI run (after 3 intra-run probe retries spaced ~12s apart). There is no per-site `consecutiveFailures` knob in `.upptimerc.yml` and no global one — verified via `upptime-monitor/src/update.ts` and `src/interfaces.ts`. So the threshold cannot be enforced inside the upstream action; it has to live in our custom downstream workflow.

**Two-axis behaviour.** Upptime auto-deletes the GitHub issue if the outage resolves within 15 minutes AND the issue has ≤1 comment (`upptime-monitor/src/summary.ts`, `skipDeleteIssues: false` default — which is what this repo runs). So for the **GitHub-issue axis**:

- Outages 0–15 min: Upptime auto-deletes the issue. No further work for us.
- Outages 15–20 min: Upptime closed the issue on recovery but didn't auto-delete (past summary.ts's 15-min cutoff). Our workflow's T+20min check sees `state == CLOSED` and explicitly `gh issue delete`'s the issue, so 15–20 min outages also leave no visible trace.
- Outages 20+ min: Issue persists as open until recovery; we page on issue-still-open at T+20min and label it `pushover-sent`.

For the **Pushover axis**, the rule is simpler: page only if the issue is still open 20 minutes after Upptime opened it.

**Token elevation for `gh issue delete`.** GitHub's `deleteIssue` GraphQL mutation requires repo-admin/maintainer scope, which the default `GITHUB_TOKEN` lacks. The workflow uses `${{ secrets.GH_PAT || secrets.GITHUB_TOKEN }}` — the same PAT Upptime's own workflows use for its `deleteIssue` mutation in `summary.ts`. If `GH_PAT` is unset (it is set in this repo), the workflow falls back to `GITHUB_TOKEN` and the delete branch will fail with a permission error in the runner log; the early-exit semantics still hold (no page, no false UP), so the failure is non-fatal — just leaves the closed issue in the repo's closed-tab.

**The label that ties open and close together.** We use a `pushover-sent` label on the issue as the single source of truth for "did we page?". The open-event branch sets it when it actually fires the Pushover; the close-event branch fires the "UP" Pushover *only* if that label is present. This keeps the two event branches stateless and survives across workflow runs (labels are issue state, not run state).

**Concurrency across services.** Each service Upptime tracks (Plex, Overseerr) gets its own issue; the workflow runs are scoped per-issue (`github.event.issue.number`), so a Plex flap and an Overseerr flap during the same 20-min window debounce independently. No mutex needed.

## Failure-handling contract

This plan has no special re-run contract. Workflow edits are inherently idempotent (the file content is the source of truth); the closeout-task rehearsal is a one-shot verification that does not mutate prod state beyond a single throwaway issue.
